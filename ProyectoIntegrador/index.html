Tecnologías de Web APIs

¿Qué es una API?

Una API (Application Programming Interface) es un conjunto de mecanismos y reglas que permiten a dos componentes de software comunicarse entre sí mediante un conjunto de definiciones y protocolos. Actúa como una capa intermedia que procesa las transferencias de datos entre sistemas.

En el contexto web, una Web API expone funcionalidades de un servidor a través del protocolo HTTP, permitiendo que cualquier cliente —web, móvil, IoT— consuma datos y servicios de forma estandarizada.

Independiente del lenguaje
Basada en HTTP/HTTPS
JSON / XML como datos
Reutilizable y escalable
// Ejemplo: consumir una Web API con fetch (JS) const response = await fetch( 'https://api.ejemplo.com/v1/usuarios', { method: 'GET', headers: { 'Authorization': 'Bearer <token>', 'Content-Type': 'application/json' } } ); const data = await response.json(); console.log(data); // ← Datos del servidor

Tipos de Web APIs

REST

El estilo más popular. Usa métodos HTTP (GET, POST, PUT, DELETE) y recursos identificados por URLs. Stateless y sin estado entre peticiones.

GraphQL

Permite al cliente pedir exactamente los datos que necesita. Desarrollado por Meta. Reduce el over-fetching y under-fetching de datos.

WebSockets

Comunicación bidireccional en tiempo real entre cliente y servidor. Ideal para chats, notificaciones push y dashboards en vivo.

SOAP

Protocolo basado en XML. Más rígido y verboso que REST, pero con mayor soporte para transacciones y seguridad empresarial.

¿Qué son los Microservicios?

Los microservicios son un enfoque arquitectónico nativo de la nube en el que una sola aplicación se conforma de muchos componentes pequeños, poco acoplados y desplegables de forma independiente. Cada servicio tiene su propia tecnología, base de datos y modelo de datos, comunicándose mediante APIs REST, eventos o agentes de mensajes.

A diferencia del monolito, donde todo vive en un único bloque, los microservicios permiten que equipos pequeños operen de forma autónoma sobre dominios acotados del negocio.

Despliegue Independiente

Cada servicio se actualiza sin afectar al resto del sistema.

Tecnología Polyglot

Cada equipo elige el lenguaje y base de datos más adecuados.

Escalado Granular

Solo escala el servicio con mayor demanda, no toda la app.

Tolerancia a Fallos

Un fallo aislado no tumba el sistema completo.

¿Por qué importan los microservicios?

Los microservicios reflejan la forma en que las organizaciones modernas quieren estructurar sus equipos: pequeños, autónomos y orientados a un producto. Esta arquitectura facilita la adopción de metodologías ágiles y la entrega continua (CI/CD), acelerando el time-to-market.

Tema 3

Desarrollo de una Web API

Recomendaciones clave

1
Definir los objetivos de la API

Antes de escribir una sola línea de código, comprende qué problemas resolverá y quiénes serán sus consumidores.

2
Seguir estándares y convenciones

Aplica los principios REST, usa JSON o XML como formato de intercambio, y nombra endpoints de forma coherente y predecible.

3
Documentar exhaustivamente

Usa herramientas como Swagger o Postman para generar documentación automática que otros desarrolladores puedan entender y usar.

4
Priorizar la seguridad

Implementa autenticación (JWT, OAuth 2.0) y autorización, valida todas las entradas y usa HTTPS en todo momento.

5
Versionar la API

Versiona tu API (/v1/, /v2/) para mantener compatibilidad hacia atrás sin romper integraciones existentes.

Proceso de desarrollo

Identifica requisitos funcionales y no funcionales, casos de uso, actores del sistema y los datos que se manipularán.

Define objetivos, asigna recursos, estima tiempos y costos, elige la metodología (Scrum, Kanban) y los frameworks del proyecto.

Define los endpoints, los métodos HTTP admitidos (GET, POST, PUT, DELETE), los parámetros de solicitud/respuesta y el esquema de autenticación. Ejemplo: /posts, /posts/{id}, /comments.

Desarrolla la API usando frameworks como Express.js (Node.js), Flask / FastAPI (Python) o Spring Boot (Java), siguiendo el diseño definido.

Realiza pruebas unitarias, de integración y de carga. Herramientas: Postman, Jest, PyTest, JMeter.

Crea documentación clara con Swagger/OpenAPI, despliega en producción (AWS, GCP, Azure) y establece un sistema de monitoreo (Prometheus, Grafana) para detectar problemas.

¿Qué es un Contenedor?

Los contenedores son paquetes ligeros de software que incluyen todo lo necesario para ejecutar una aplicación en cualquier entorno: código, runtime, bibliotecas y configuración. Al virtualizar el sistema operativo en lugar del hardware, son mucho más eficientes que las máquinas virtuales.

Google usa contenedores para todo: Gmail, YouTube, la Búsqueda. La tecnología evolucionó hasta convertirse en Docker (contenedores) y Kubernetes (orquestación), las herramientas estándar de la industria.

Docker — El estándar de contenedores

Docker es la plataforma más usada para crear y ejecutar contenedores. Con un simple Dockerfile defines el entorno de tu aplicación y la empaquetas de forma reproducible.

# Dockerfile de ejemplo — Node.js API FROM node:20-alpine WORKDIR /app COPY package*.json ./ RUN npm install --production COPY . . EXPOSE 3000 CMD ["node", "index.js"]
# Comandos Docker esenciales docker build -t mi-api:v1 . docker run -p 3000:3000 mi-api:v1 docker ps # contenedores activos docker-compose up -d # multi-contenedor

Kubernetes — Orquestación de contenedores

Arquitectura de clúster

Un clúster Kubernetes tiene un Nodo Control Plane que gestiona el estado deseado del sistema y múltiples Nodos Worker que ejecutan los contenedores.

Pods y Deployments

El Pod es la unidad mínima de Kubernetes (1+ contenedores). Un Deployment gestiona replicas y rollbacks automáticos de pods.

Auto-Scaling

Kubernetes puede escalar automáticamente el número de pods según la demanda (CPU, memoria), garantizando disponibilidad sin desperdicio de recursos.

Secrets & ConfigMaps

Gestiona credenciales y configuraciones de forma segura sin hardcodear valores sensibles en las imágenes de los contenedores.

Persistent Volumes

Permite que los contenedores almacenen datos que persisten más allá del ciclo de vida del pod, con soporte para múltiples proveedores de almacenamiento.

Servicios & Ingress

Los Services exponen pods como endpoints de red. El Ingress gestiona el tráfico externo con soporte de dominio y HTTPS.

¿Por qué los contenedores revolucionaron el desarrollo?

Los contenedores resolvieron el clásico problema de "en mi máquina funciona". Al empaquetar la aplicación junto con todas sus dependencias, garantizan que el comportamiento sea idéntico en cualquier entorno: laptop del desarrollador, servidor de staging o producción en la nube.

¿Qué es el Cloud Computing?

El cómputo en la nube es la entrega de recursos informáticos —servidores, almacenamiento, bases de datos, redes, software e inteligencia— a través de internet ("la nube") bajo un modelo de pago por uso. Elimina la necesidad de adquirir y mantener infraestructura física.

Una aplicación nativa de la nube (cloud-native) es aquella diseñada desde cero para aprovechar las ventajas del cloud: escalado elástico, alta disponibilidad, despliegue automatizado y resiliencia ante fallos.

Modelos de servicio
IaaS — Infraestructura como Servicio

El proveedor ofrece infraestructura virtualizada (VMs, redes, almacenamiento). Tú controlas el SO y las aplicaciones. Ej: AWS EC2, Azure VMs, Google Compute Engine.

PaaS — Plataforma como Servicio

El proveedor gestiona la infraestructura y el SO. Tú solo despliegas tu código. Ej: Heroku, Google App Engine, AWS Elastic Beanstalk.

SaaS — Software como Servicio

El proveedor gestiona todo. Tú solo usas la aplicación vía navegador. Ej: Gmail, Salesforce, Microsoft 365, Slack.

FaaS — Funciones como Servicio (Serverless)

Ejecutas código en respuesta a eventos sin gestionar servidores. Solo pagas por ejecución. Ej: AWS Lambda, Google Cloud Functions, Azure Functions.

¿Por qué migrar a la nube?

La nube no es solo infraestructura: es un cambio de paradigma. Permite a equipos pequeños construir y operar aplicaciones que compiten con grandes empresas, sin necesidad de invertir en servidores físicos. La combinación de microservicios, contenedores y servicios cloud es la base del desarrollo moderno.

Glosario — Unidad II
API Gateway

Punto de entrada único para todos los microservicios. Gestiona autenticación, rate-limiting y enrutamiento.

Circuit Breaker

Patrón de resiliencia que detiene llamadas a servicios fallidos para evitar cascadas de errores.

Service Discovery

Mecanismo por el cual los servicios se registran y se localizan dinámicamente (Consul, Eureka, K8s DNS).

Orquestación

Gestión automatizada del ciclo de vida de contenedores: despliegue, escalado, networking y almacenamiento.

JWT / OAuth 2.0

Estándares de autenticación y autorización para APIs. JWT = tokens firmados. OAuth = delegación de permisos.

Serverless

Modelo donde el código se ejecuta en funciones sin gestionar servidores. Pago por ejecución (AWS Lambda).
