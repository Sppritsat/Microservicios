Tema 1

Tecnologías de Web APIs

¿Qué es una API?

Una API (Application Programming Interface) es un conjunto de mecanismos y reglas que permiten a dos componentes de software comunicarse entre sí mediante definiciones y protocolos estandarizados. Actúa como una capa intermedia que procesa las transferencias de datos entre sistemas, permitiendo a las empresas abrir sus datos y funcionalidades a desarrolladores externos, socios comerciales y departamentos internos. (IBM, 2023)

En el contexto web, una Web API expone esas funcionalidades a través del protocolo HTTP/HTTPS, de modo que cualquier cliente —navegador, app móvil, dispositivo IoT o incluso otro servidor— puede consumir datos y servicios de forma estandarizada sin importar el lenguaje de programación que use.

Las APIs son el pegamento invisible de internet moderno. Cuando usas Google Maps dentro de otra app, cuando pagas con PayPal en un sitio externo, o cuando ves un video de YouTube embebido en una página, estás consumiendo una Web API.

Tipos de Web APIs

Existen varios estilos arquitectónicos para diseñar Web APIs. La elección depende del caso de uso, los requisitos de rendimiento y la experiencia del equipo de desarrollo.

REST

El estilo más popular. Usa métodos HTTP estándar (GET, POST, PUT, DELETE) y recursos identificados por URLs únicas. Es stateless: cada petición contiene toda la información necesaria. Ideal para APIs públicas y servicios web de propósito general.

GraphQL

Desarrollado por Meta en 2015. Permite al cliente especificar exactamente qué datos necesita en una sola petición, evitando el over-fetching y under-fetching. Un único endpoint /graphql maneja todas las consultas y mutaciones.

WebSockets

Protocolo de comunicación bidireccional y persistente entre cliente y servidor. Ideal para aplicaciones en tiempo real: chats, notificaciones push, dashboards en vivo, juegos multijugador y cotizaciones financieras.

SOAP

Protocolo basado en mensajes XML. Más rígido que REST pero con mayor soporte para transacciones distribuidas (ACID), seguridad empresarial (WS-Security) y contratos formales mediante WSDL. Aún prevalece en banca y sistemas legacy.

¿Cómo funciona una API REST?

REST (Representational State Transfer) fue definido por Roy Fielding en su disertación doctoral del año 2000. Se basa en seis principios arquitectónicos: interfaz uniforme, sin estado, caché, sistema en capas, código bajo demanda (opcional) y arquitectura cliente-servidor.

Cada recurso se identifica con una URL única y se manipula usando los verbos estándar de HTTP:

¿Por qué las APIs son el futuro?

Las APIs transformaron la industria al permitir que empresas como Stripe, Twilio o SendGrid construyeran negocios enteros vendiendo acceso a sus APIs. Este modelo —conocido como API-first o API economy— es ahora estándar: las empresas diseñan sus productos como plataformas desde el inicio, exponiendo funcionalidades que terceros pueden integrar para crear valor adicional.

La estandarización mediante OpenAPI (Swagger), la seguridad con OAuth 2.0 y JWT, y las herramientas de gestión como Kong o AWS API Gateway han consolidado a las Web APIs como la columna vertebral del software moderno.

¿Qué son los Microservicios?

Los microservicios —o arquitectura de microservicios— son un enfoque arquitectónico nativo de la nube en el que una sola aplicación se construye como un conjunto de muchos componentes pequeños, poco acoplados y desplegables de forma independiente. Cada servicio se organiza alrededor de una capacidad de negocio concreta y se comunica con los demás mediante APIs REST, transmisión de eventos o agentes de mensajes como Kafka o RabbitMQ. (IBM, 2023)

Este enfoque permite que equipos pequeños y autónomos —siguiendo la "regla de las dos pizzas" de Amazon— operen sobre dominios acotados del negocio sin depender del ciclo de despliegue global de la organización.

Los microservicios no solo son una decisión técnica: son un modelo organizativo. La arquitectura refleja la estructura del equipo (Ley de Conway) y facilita metodologías ágiles, CI/CD y una cultura DevOps genuina.

Microservicios vs. otras arquitecturas

La diferencia más importante con una arquitectura monolítica es que en el monolito toda la lógica de negocio, la capa de datos y la interfaz viven juntos en una única unidad desplegable. Cualquier cambio, por pequeño que sea, requiere compilar y desplegar toda la aplicación.

Características y patrones clave

Un sistema bien diseñado de microservicios incorpora varios patrones arquitectónicos que garantizan su correcto funcionamiento a escala.

Desacoplamiento

Cada servicio tiene su propio ciclo de vida, base de datos y modelo de datos. Los cambios en uno no fuerzan cambios en los demás. El límite entre servicios se denomina Bounded Context (DDD).

API Gateway

Punto de entrada único que enruta solicitudes, gestiona autenticación (JWT / OAuth), aplica rate limiting, transforma formatos y agrega respuestas de múltiples servicios en una sola.

Circuit Breaker

Cuando un servicio falla repetidamente, el circuit breaker "abre" y deja de enviarle tráfico temporalmente, devolviendo respuestas de respaldo y evitando cascadas de fallos.

Event-Driven

Los servicios publican eventos en un bus (Kafka, RabbitMQ) y otros los consumen asincrónicamente. Reduce el acoplamiento directo y mejora la resiliencia del sistema ante picos de carga.

Service Discovery

Los servicios se registran dinámicamente (Consul, Eureka, Kubernetes DNS) y se localizan sin necesidad de IPs hardcodeadas, facilitando el escalado horizontal automático.

Observabilidad

Los tres pilares: Métricas (Prometheus), Logs (ELK Stack) y Trazas distribuidas (Jaeger, Zipkin). Imprescindibles para depurar sistemas distribuidos.

Casos de éxito en la industria

Netflix

Migró de un monolito a más de 700 microservicios. Cada servicio (catálogo, reproducción, recomendaciones, facturación) escala de forma independiente para atender a 270M+ de usuarios.

Amazon

Pionero de la arquitectura. Obligó a todos sus equipos a exponer sus funcionalidades como APIs internas (la famosa memo de Bezos del 2002), lo que luego se convirtió en AWS.

Uber

Separó sus servicios de viajes, pagos, mapas, notificaciones y precios en microservicios independientes, permitiendo escalar cada área según la demanda geográfica en tiempo real.

Tema 3

Desarrollo de una Web API

Recomendaciones clave antes de desarrollar

El éxito de una API no depende únicamente de su código, sino del proceso previo al desarrollo. Estas recomendaciones, basadas en el material del curso, son fundamentales para construir APIs robustas, usables y mantenibles.

1
Definir los objetivos de la API

Antes de escribir una sola línea de código, comprende qué problema resolverá, quiénes serán sus consumidores (web, móvil, terceros) y qué datos o funcionalidades expondrá. Esta claridad evita rediseños costosos.

2
Seguir estándares y convenciones

Aplica los principios REST: URLs en sustantivos plurales (/usuarios), verbos HTTP correctos, códigos de estado HTTP semánticos (200, 201, 400, 401, 404, 500) y JSON como formato de intercambio.

3
Documentar exhaustivamente

Usa Swagger / OpenAPI para generar documentación interactiva automáticamente. Una API bien documentada es una API que los desarrolladores adoptan. Postman complementa para pruebas y ejemplos de uso.

4
Priorizar la seguridad

Implementa autenticación (JWT, OAuth 2.0) y autorización por roles. Valida y sanitiza todos los inputs, usa HTTPS en producción, aplica rate limiting y registra todos los accesos (audit logs).

5
Versionar la API

Versiona desde el inicio (/api/v1/) para poder introducir cambios sin romper integraciones existentes. Mantén al menos la versión anterior activa durante un período de transición (deprecation policy).

Proceso de desarrollo paso a paso

El desarrollo de una Web API sigue un ciclo estructurado que garantiza la calidad, la seguridad y la mantenibilidad del producto final.

1
Análisis de requerimientos

Identifica requisitos funcionales (qué hará la API) y no funcionales (rendimiento, seguridad, disponibilidad). Define los casos de uso, actores del sistema y los datos que se manipularán.

2
Planificación

Define objetivos medibles, asigna recursos, estima tiempos y costos, elige la metodología (Scrum, Kanban) y los frameworks del proyecto (Express.js, FastAPI, Spring Boot).

3
Diseño de la API

Define los endpoints, métodos HTTP admitidos, parámetros de solicitud y respuesta, esquema de autenticación y el modelo de datos. Documenta antes de implementar (API-first design).

4
Implementación

Desarrolla la lógica de negocio siguiendo el diseño. Usa frameworks maduros: Express.js (Node), FastAPI / Flask (Python), Spring Boot (Java), Laravel (PHP).

5
Pruebas

Pruebas unitarias (Jest, PyTest), de integración, de carga (JMeter, k6) y de seguridad (OWASP ZAP). Automatiza las pruebas en el pipeline CI/CD para detectar regresiones tempranamente.

6–9
Despliegue, Monitoreo y Publicación

Despliega en producción (AWS, GCP, Azure), establece monitoreo (Prometheus, Grafana), realiza mantenimiento continuo y opcionalmente publica la API en un marketplace (RapidAPI, API Gateway).

¿Qué es un Contenedor?

Los contenedores son paquetes ligeros de software que incluyen todo lo necesario para ejecutar una aplicación en cualquier entorno: código fuente, runtime, bibliotecas, variables de entorno y archivos de configuración. Al virtualizar el sistema operativo en lugar del hardware completo, son mucho más eficientes que las máquinas virtuales tradicionales. (Google Cloud, 2024)

Google ha usado contenedores durante más de una década para ejecutar Gmail, YouTube y la Búsqueda. Esta experiencia derivó en la creación de Kubernetes, que Google donó a la CNCF en 2014 y que hoy es el estándar de orquestación de contenedores de la industria.

La gran ventaja de los contenedores es que resuelven el clásico problema de "en mi máquina funciona": al empaquetar la aplicación junto con todas sus dependencias, garantizan un comportamiento idéntico en cualquier entorno de ejecución.

Docker — El estándar de contenedores

Docker es la plataforma más utilizada para crear, distribuir y ejecutar contenedores. Con un Dockerfile defines el entorno de tu aplicación de forma reproducible. Docker Hub es el registro público de imágenes con millones de imágenes listas para usar.

Kubernetes — Orquestación a escala

Kubernetes (K8s) automatiza el despliegue, escalado y gestión de aplicaciones en contenedores. Su arquitectura se basa en un Control Plane que gestiona el estado deseado del sistema y múltiples Nodos Worker que ejecutan los workloads.

Pods y Deployments

El Pod es la unidad mínima de Kubernetes (1 o más contenedores). Un Deployment gestiona réplicas, rolling updates y rollbacks automáticos.

Horizontal Pod Autoscaler

K8s escala automáticamente el número de pods según métricas de CPU, memoria o métricas personalizadas, garantizando disponibilidad sin desperdicio de recursos.

Services e Ingress

Los Services exponen pods como endpoints de red estables. El Ingress gestiona el tráfico externo HTTP/S con soporte de dominio, TLS y reglas de enrutamiento.

Secrets y ConfigMaps

Gestiona credenciales y configuraciones de forma segura sin hardcodear valores sensibles en las imágenes. Los Secrets están codificados en Base64 y pueden cifrarse en reposo.

Persistent Volumes

Permite almacenar datos que persisten más allá del ciclo de vida de un pod. Compatible con múltiples backends: AWS EBS, Google Persistent Disk, NFS, local storage.

Namespaces y RBAC

Los Namespaces permiten aislar recursos en un clúster compartido. RBAC (Control de Acceso Basado en Roles) define qué usuarios o pods pueden hacer qué operaciones.

Docker + Kubernetes: la combinación ganadora

Docker empaqueta la aplicación; Kubernetes la opera a escala. Juntos forman la base del desarrollo moderno cloud-native. La mayoría de los proveedores cloud ofrecen Kubernetes gestionado: EKS (AWS), GKE (Google) y AKS (Azure), eliminando la complejidad de operar el Control Plane.

¿Qué es el Cloud Computing?

El cómputo en la nube es la entrega de recursos informáticos —servidores, almacenamiento, bases de datos, redes, software e inteligencia artificial— a través de internet bajo un modelo de pago por uso. Elimina la necesidad de adquirir, mantener y actualizar infraestructura física propia.

Una aplicación nativa de la nube (cloud-native) es aquella diseñada desde cero para aprovechar las ventajas del cloud: elasticidad, alta disponibilidad global, despliegue automatizado, resiliencia ante fallos y escalado horizontal ilimitado.

El cloud no es solo una cuestión tecnológica: es un cambio de paradigma cultural. Permite a equipos pequeños construir y operar aplicaciones que compiten en escala con grandes corporaciones, sin inversión inicial en infraestructura (CapEx → OpEx).

Modelos de servicio en la nube

Cada modelo ofrece un nivel diferente de abstracción y control. La elección depende del equipo, el presupuesto y la complejidad de la aplicación.

IaaS

Infraestructura como Servicio. El proveedor ofrece VMs, redes y almacenamiento. Tú controlas el SO, middleware y aplicaciones. Máximo control, máxima responsabilidad.
AWS EC2, Google Compute Engine, Azure VMs

PaaS

Plataforma como Servicio. El proveedor gestiona la infraestructura y el SO. Tú solo despliegas tu código. Ideal para equipos que quieren enfocarse en el producto.
Heroku, Google App Engine, AWS Elastic Beanstalk

SaaS

Software como Servicio. El proveedor gestiona todo. Tú accedes a la aplicación vía navegador o API. Sin instalación, sin mantenimiento.
Gmail, Salesforce, Microsoft 365, Slack, Zoom

FaaS / Serverless

Funciones como Servicio. Ejecutas código en respuesta a eventos sin gestionar servidores. Pagas solo por ejecución (milisegundos). Escala a cero automáticamente.
AWS Lambda, Google Cloud Functions, Azure Functions

La convergencia: Microservicios + Contenedores + Nube

Glosario — Unidad II
API Gateway

Punto de entrada único para todos los microservicios. Gestiona autenticación, rate-limiting y enrutamiento.

Circuit Breaker

Patrón de resiliencia que detiene llamadas a servicios fallidos para evitar cascadas de errores.

Service Discovery

Mecanismo por el cual los servicios se registran y localizan dinámicamente (Consul, Eureka, K8s DNS).

Orquestación

Gestión automatizada del ciclo de vida de contenedores: despliegue, escalado, networking y almacenamiento.

JWT / OAuth 2.0

Estándares de autenticación y autorización. JWT = tokens firmados. OAuth = delegación de permisos entre apps.

Serverless / FaaS

Ejecución de código en funciones sin gestionar servidores. Pago por milisegundo de ejecución (AWS Lambda).

OpenAPI / Swagger

Especificación estándar para describir APIs RESTful. Swagger UI genera documentación interactiva automáticamente.

CI/CD

Integración Continua y Entrega Continua. Pipelines automáticos que prueban y despliegan código en cada commit.
